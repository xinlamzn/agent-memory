=== Extraction Improvements

[[schema-auto-detection]]
==== Schema Auto-Detection

[horizontal]
Priority:: Low
Current State:: Users must manually select a domain schema for entity extraction.

**Suggestion**: Add automatic schema detection that analyzes input text and selects the most appropriate domain schema.

.Implementation Ideas
* Use text classification to identify document type
* Support confidence scores and fallback to generic POLEO schema
* Allow schema blending for mixed-domain content

'''

[[confidence-calibration]]
==== Entity Confidence Calibration

[horizontal]
Priority:: Low
Current State:: Confidence scores from different extractors are not calibrated against each other.

**Suggestion**: Implement confidence score calibration so that scores from spaCy, GLiNER, and LLM extractors are comparable.

.Implementation Ideas
* Use a calibration dataset with ground truth
* Apply Platt scaling or isotonic regression
* Normalize scores to a common scale

'''

[[result-caching]]
==== Extraction Result Caching

[horizontal]
Priority:: Low
Current State:: No caching of extraction results.

**Suggestion**: Add optional caching layer for extraction results keyed by content hash.

.Benefits
* Avoid reprocessing identical content
* Speed up re-runs and testing
* Reduce API costs for LLM extraction

'''

=== API & Developer Experience

[[pipeline-dsl]]
==== Extraction Pipeline Builder DSL

[horizontal]
Priority:: Low
Current State:: `ExtractorBuilder` provides a fluent API but is somewhat verbose.

**Suggestion**: Create a more declarative DSL for pipeline configuration.

.Example
[source,python]
----
pipeline = Pipeline.from_config("""
stages:
  - spacy:
      model: en_core_web_sm
  - gliner:
      schema: podcast
      threshold: 0.5
  - llm:
      model: gpt-4o-mini
      extract_relations: true
merge: confidence
""")
----

'''

==== Schema Validation and Testing

[horizontal]
Priority:: Low

**Suggestion**: Add a schema validation tool that tests extraction against sample documents.

[[temporal-versioning]]

[horizontal]
Priority:: Low

.Implementation Ideas
* Add `valid_from` and `valid_to` timestamps

* Support point-in-time queries

== v0.0.2 Roadmap (Implemented & Documented)

The following features were implemented and documented as part of the v0.0.2 release cycle.

=== Entity Extraction

[[batch-extraction]]
==== Batch Extraction API ✅

[horizontal]
Priority:: High
Status:: **IMPLEMENTED & DOCUMENTED**
Implementation:: `ExtractionPipeline.extract_batch()` and `GLiNEREntityExtractor.extract_batch()`
Documentation:: xref:how-to/batch-processing.adoc[Batch Processing Guide]

**Features Implemented**:

* `extract_batch()` method on `ExtractionPipeline` for parallel processing of multiple texts
* `extract_batch()` method on `GLiNEREntityExtractor` using native batch inference for GPU efficiency
* `BatchExtractionResult` dataclass with aggregate statistics
* Progress callback support via `on_progress: Callable[[int, int], None]`
* Configurable concurrency and batch size
* Error handling options: `fail_fast=True` to stop on first error

'''

[[streaming-extraction]]
==== Streaming Extraction for Long Documents ✅

[horizontal]
Priority:: Medium
Status:: **IMPLEMENTED & DOCUMENTED**
Implementation:: `StreamingExtractor` class in `neo4j_agent_memory.extraction.streaming`
Documentation:: xref:how-to/batch-processing.adoc#streaming[Streaming Extraction Guide]

**Features Implemented**:

* `StreamingExtractor` class wrapping any `EntityExtractor` for chunked processing
* `extract_streaming()` async generator that yields `StreamingChunkResult` for each chunk
* Chunking strategies: character-based and token-based
* Automatic entity deduplication across chunks
* Progress callback support for UI integration

'''

[[gliner-relations]]
==== GLiREL Relation Extraction ✅

[horizontal]
Priority:: Medium
Status:: **IMPLEMENTED & DOCUMENTED**
Implementation:: `GLiRELExtractor` and `GLiNERWithRelationsExtractor`
Documentation:: xref:reference/extractors.adoc[Extractors Reference]

**Features Implemented**:

* `GLiRELExtractor` class for standalone relation extraction
* `GLiNERWithRelationsExtractor` combining GLiNER + GLiREL
* `DEFAULT_RELATION_TYPES` with 20+ POLE+O-optimized relation types
* Zero API costs (no LLM calls), full offline support

'''
