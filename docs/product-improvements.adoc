[[entity-dedup-management-api]]
==== Entity Deduplication Management API

[horizontal]
Priority:: High
Current State:: The library creates `SAME_AS` relationships and supports deduplication on ingest, but there is no public API to find, review, or merge duplicate entities after the fact.

**Suggestion**: Add management methods to `LongTermMemory` for human-in-the-loop entity resolution.

.Proposed Methods
* `find_potential_duplicates(entity_type, status, limit)` -- list entity pairs flagged as potential duplicates
* `merge_duplicate_entities(source_id, target_id)` -- merge source into target, transferring all relationships
* `review_duplicate(entity1_id, entity2_id, decision)` -- confirm or reject a duplicate pair
* `get_deduplication_stats()` -- aggregate stats on pending/confirmed/rejected duplicates

'''

=== Bug Fixes & Technical Debt

[[store-extracted-relations]]
==== Store Extracted Relations in Short-Term Memory

[horizontal]
Priority:: High
Current State:: The entity extraction pipeline extracts both entities and relations, but only entities are stored. Relations in `ExtractionResult.relations` are silently discarded.

**Problem**: In `ShortTermMemory._extract_and_link_entities()` and `extract_entities_from_session()`, only `result.entities` is processed. `result.relations` is never used, so `RELATED_TO` relationships are never created between entities. Entity-to-entity relationships like "Brian Chesky FOUNDED Airbnb" are lost.

**Suggested Fix**: After creating entity nodes, iterate over `result.relations`, resolve source/target entity IDs, and call `LongTermMemory.add_relationship()` to create the inter-entity relationships.

'''

[[apoc-free-queries]]
==== APOC-Free Queries

[horizontal]
Priority:: High
Current State:: `_build_metadata_filter_clause()` uses `apoc.convert.fromJsonMap()` which requires the APOC plugin and fails on Neo4j Aura.

**Suggestion**: Audit all Cypher queries for APOC usage and provide pure-Cypher fallbacks or use JSON string operations. Document any remaining APOC requirements clearly.

'''

[[graceful-missing-relationship-types]]
==== Graceful Handling of Missing Relationship Types

[horizontal]
Priority:: Low
Current State:: `get_graph()` queries for `RELATED_TO` relationships between entities, generating Neo4j warnings when the relationship type does not exist in the database.

**Suggestion**: Either query `db.relationshipTypes()` first and only include existing types, use a `WHERE type(r) IN [...]` pattern that avoids warnings, or add a configurable `entity_relationship_types` parameter to `get_graph()`.

'''

=== Framework API Stability

These improvements address challenges discovered during the Microsoft Agent Framework integration, where the ChatMessage API used `text` instead of the expected `content` property.

[[api-version-detection]]
==== Framework API Version Detection

[horizontal]
Priority:: High
Status:: Concept
Discovered:: During Microsoft Agent Framework integration (January 2026)

Automatically detect framework API versions and adapt behavior accordingly.

.Problem Observed
The Microsoft Agent Framework's `ChatMessage` class uses `.text` property for message content instead of the more common `.content` property. This caused integration failures when the code assumed `.content` was available.

.Implementation Ideas
* Introspect framework classes at import time to detect available properties
* Maintain version compatibility matrices in code
* Runtime warnings when using deprecated or changed APIs
* Automatic property mapping based on detected version

.Example Implementation
[source,python]
----
def _get_message_content(self, msg: Any) -> str | None:
    """Extract content with automatic API detection."""
    # Detect which property the message type uses
    if hasattr(msg, "text") and msg.text:  # Microsoft Agent Framework
        return msg.text
    if hasattr(msg, "content"):  # OpenAI, LangChain, etc.
        return msg.content
    if isinstance(msg, dict):
        return msg.get("content") or msg.get("text")
    return None
----

'''

[[message-format-abstraction]]
==== Message Format Abstraction Layer

[horizontal]
Priority:: Medium
Status:: Concept
Discovered:: During Microsoft Agent Framework integration (January 2026)

Create a unified message abstraction that normalizes different framework message formats.

.Problem Observed
Each AI framework has its own message format:

* OpenAI: `{"role": "user", "content": "..."}`
* Microsoft Agent Framework: `ChatMessage(role="user", text="...")`
* LangChain: `HumanMessage(content="...")`
* Anthropic: `{"role": "user", "content": [{"type": "text", "text": "..."}]}`

.Implementation Ideas
* Define a `UnifiedMessage` protocol/interface
* Create adapter functions for each framework
* Support round-trip conversion (unified -> framework-specific)
* Handle multimodal content (text, images, tool calls)

.Proposed Interface
[source,python]
----
from typing import Protocol

class UnifiedMessage(Protocol):
    @property
    def role(self) -> str: ...

    @property
    def content(self) -> str: ...

    @property
    def tool_calls(self) -> list[dict] | None: ...

    @classmethod
    def from_openai(cls, msg: dict) -> "UnifiedMessage": ...

    @classmethod
    def from_microsoft(cls, msg: "ChatMessage") -> "UnifiedMessage": ...

    def to_openai(self) -> dict: ...

    def to_microsoft(self) -> "ChatMessage": ...
----

'''

[[framework-adapter-testing]]
==== Framework Adapter Compatibility Testing

[horizontal]
Priority:: High
Status:: Concept
Discovered:: During Microsoft Agent Framework integration (January 2026)

Automated testing infrastructure to detect breaking API changes in dependent frameworks.

.Problem Observed
Framework APIs can change between versions (especially in beta releases like Microsoft Agent Framework). Tests that mock the framework classes may pass even when the actual framework API has changed.

.Implementation Ideas
* Dedicated CI job that installs actual framework packages (not mocks)
* API surface tests that verify expected methods/properties exist
* Canary tests that run against framework nightly builds
* Automated alerts when framework releases new versions

.Test Categories
[source,python]
----
# API existence tests (should run with real framework installed)
def test_chat_message_has_expected_properties():
    """Verify ChatMessage API matches our expectations."""
    from agent_framework import ChatMessage

    msg = ChatMessage(role="user", text="Hello")

    # Verify expected properties exist
    assert hasattr(msg, "role"), "ChatMessage missing 'role' property"
    assert hasattr(msg, "text"), "ChatMessage missing 'text' property"

    # Verify deprecated properties are NOT used in our code
    # assert not hasattr(msg, "content"), "ChatMessage has 'content' - update our code"

# Version compatibility tests
def test_framework_version_supported():
    """Verify installed framework version is in supported range."""
    import agent_framework
    from packaging import version

    installed = version.parse(agent_framework.__version__)
    min_supported = version.parse("1.0.0b251001")
    max_tested = version.parse("1.0.0b251223")

    assert installed >= min_supported, f"Framework {installed} < minimum {min_supported}"
    if installed > max_tested:
        warnings.warn(f"Framework {installed} > tested {max_tested}, may have API changes")
----

[[bind-tool-pattern]]
==== _bind_tool ADK Function Signature Pattern ✅

[horizontal]
Priority:: Medium
Status:: **IMPLEMENTED**
Implementation:: `examples/google-cloud/financial-advisor/backend/src/agents/kyc_agent.py` (and all agent modules)

**Features Implemented**:

* `_bind_tool()` helper that injects `neo4j_service` into tool functions while hiding it from ADK's `FunctionTool` signature inspection
* Uses `inspect.signature()` to build a new signature excluding the `neo4j_service` parameter
* Uses `functools.wraps` to preserve the original function's name and docstring (which ADK uses for tool descriptions)
* Solves the problem of ADK treating all function parameters as agent-callable — services should not be exposed to the LLM

.Pattern
[source,python]
----
def _bind_tool(func, neo4j_service):
    """Bind neo4j_service to a tool function, hiding it from ADK."""
    @wraps(func)
    async def wrapper(**kwargs):
        return await func(neo4j_service=neo4j_service, **kwargs)

    # Remove neo4j_service from the visible signature
    sig = inspect.signature(func)
    params = [p for name, p in sig.parameters.items() if name != "neo4j_service"]
    wrapper.__signature__ = sig.replace(parameters=params)
    return wrapper
----
